hash路由和浏览器路由的选择

浏览器路由和hash路由是有区别的，浏览器路由顾明思意就是以前后台服务器配置的那种路由，/路由名。hash路由则是和完全不同的一种路由，是用锚点实现的一种路由，所以它的性质就基本和锚点的性质差不多 （我在做前端路由的时候一般都会选择hash路由，尤其是在做微信公众号时，如果用浏览器路由会出现路径问题， 具体什么问题就记不清了）。

fetch请求需要牢记的两点

  • 请求返回一个Promise对象

    当接收到一个代表错误的 HTTP 状态码时，从 fetch()返回的 Promise 不会被标记为 reject， 即使该 HTTP 响应的状态码是 404 或 500。相反，它会将 Promise 状态标记为 resolve （但是会将 resolve 的返回值的 ok 属性设置为 false ），仅当网络故障时或请求被阻止时，才会标记为 reject

  • 默认情况下，fetch 不会从服务端发送或接收任何 cookies, 如果站点依赖于用户 session，则会导致未经认证的请求（要发送 cookies，必须设置 credentials 选项）。

SPA中的页面跳转的传参

在一个大型项目中会有很多页面跳转，我们暂且把页面跳转简单的分为两种：

  1. 无返回状态的跳转：
    像产品的详情、订单详情、订单列表等这类页面，返回上一页不需要当前页面的参数。只需要执行，window.history.go(-1)
  2. 有返回状态跳转：
    像地址列表，新建地址等页面，产品为了提高用户体验，一般要求选择完地址后返回上一页的时候要带有选中的地址信息（当然你也可以把选择订单的页面单独拿出来做），以及从下单页面选择地址时进入地址列表为空时，需要直接进入新建地址页面，新建地址完成后也要直接返回下单页面。这里的地址列表（假如选择地址和地址列表为一个页面）和新建地址页面操作完成后需要带着当前页面的信息返回。

页面跳转到下一页传参也有两种方法：

  1. 通过框架的状态管理机制传参：
    这种方法操作相对简单，无需配置路由，直接把要传的参数存储在状态里面。因为状态会随着页面的刷新状态就会变成初始化状态，所以如果当前页面的内容依赖上一个页面的状态的时候，刷新当前页面这个状态就会变成初始值，页面分享的时候甚至可能找不到这个状态。这种页面之间的交互是不牢固的，我一般不会使用这种不靠谱的方式
  2. 通过路由传参：
    通过路由传参的方式可能相对于上一种方式要稍微复杂一点，他需要去配置对应的路由。但是通过路由传参跳转页面，参数是存储在url中的，也就不会出现通过状态传递参数的方式的种种问题。

参数回传：

有返回状态页面跳转除了要把前一个页面的参数原封不动的还给他，还要当前页面的处理结果也传递给前一个页面。
假如我的立即下单页的路径是 #/createOrder/01234 (默认使用路由传参，01234为商品id)，在立即下单页面需要选择地址，选择地址页的路径是#/address/chance(假如选择地址和地址列表为同一个页面，chance用来标记当前页是选择地址页)。如果#/createOrder页面#/address页面传回来的地址那也就是我们上文所说的无返回状态的页面跳转，直接使用window.history.go(-1)直接返回就可以了，上一个页面的路径不会发生任何改变。这里其实又可以分为两种情况，这里我们只考虑最复杂的那种情况，就是#/createOrder页面也是依赖#/address页的状态的，也就是说产品需要，当#/address页面一旦做出了选择，#/createOrder页面就要记住这个选择，不管是刷新还是分享都要记住。使用window.history.go(-1)就无法把#/address页面做出的选择传递到#/createOrder页面中去。解决这个问题我就直接把#/createOrder页面中的参数也传递到#/address页面中去，选择完地址后再原封不动的把参数返回，如果一个#/createOrder页面的的参数有很多，我们要尽量避免给#/address页面添加太多的参数，我们只需给#/address页面url添加一个参数字段（beforeParams），然后把从上一个页面那个过来的参数转换成字符串，存在beforeParams这个字段里面，解析的时候在把这个字符串传换成对象，通过window.location.href跳转回上一个页面。
